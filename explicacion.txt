He leído el PDF del proyecto. ft_irc consiste en desarrollar tu propio Servidor IRC (Internet Relay Chat) utilizando C++ 98.

Aquí tienes un resumen claro de lo que debes hacer y sus reglas principales:

🎯 Objetivo Principal
Crear un servidor capaz de aceptar múltiples conexiones de clientes (usando un cliente 
IRC real, como irssi o HexChat) de forma simultánea, permitiendo a los usuarios 
intercambiar mensajes en tiempo real, unirse a canales y enviarse mensajes privados.

⚙️ Reglas Técnicas Obligatorias
Un solo hilo / No-Blocking: Está prohibido utilizar fork() o hilos. Todo el programa de 
lectura y escritura debe ser asíncrono y no bloqueante (non-blocking).
Uso de poll() (o similar): Solo puedes usar una única instancia de poll() (o epoll, 
select, kqueue) para gestionar todas tus operaciones de Entrada/Salida (lectura, 
escritura, escucha, etc.). Hacer lectura/escritura (recv/send) sin pasar por esta 
comprobación equivale a una nota de 0.
Robustez ante paquetes rotos: Si un cliente envía un comando en varios trozos sueltos por 
problemas de latencia (ejemplo al enviar "com", luego "man", luego "d\n"), el servidor 
debe almacenarlos y ensamblar el comando completo antes de ejecutarlo.
Prohibido el uso de librerías externas como Boost.
🗨️ Funciones y Comandos a Implementar
Debes permitir a los clientes:

Autenticarse con una contraseña en el servidor y configurar su nickname y username.
Enviar y recibir mensajes privados y canales grupales.
Diferenciar entre usuarios normales y Operadores de un canal.
Comandos específicos para los Operadores de Canal:

KICK: Expulsar usuarios del canal.
INVITE: Invitar a un usuario a tu canal.
TOPIC: Poner o ver el título/tema del canal.
MODE: Cambiar la configuración del canal. Tienes que implementar los siguientes modos:
i: Canal sólo por invitación.
t: Sólo operadores configuran el TOPIC.
k: Poner contraseña al canal.
o: Dar/quitar privilegios de operador a otro usuario.
l: Limitar el número máximo de usuarios en el canal.

resumen de lo que he programado que cumple exactamente con los requisitos del subject:

_initSocket():

Crea el socket principal con socket().
Permite reutilizar el puerto rápidamente con setsockopt(SO_REUSEADDR).
Importantísimo: Convierte el socket a no-bloqueante usando fcntl(fd, F_SETFL, O_NONBLOCK);.
Usa bind() para conectarlo al puerto especificado y listen() para ponerlo a escuchar.
Lo añade a un vector _pollFds (con el evento POLLIN, que significa "notifícame cuando 
haya cosas para leer").
start() (el bucle poll()):

Inicia un bucle de vida libre while(true).
Usa el famoso poll() para esperar indefinidamente (o ejecutar instantáneamente) cualquier 
evento nuevo.
Dentro, recorre el vector de file descriptors:
Si la actividad es sobre el socket principal, ejecuta _acceptNewClient().
Si la actividad es sobre el socket de un cliente, ejecuta _handleClientData().
_acceptNewClient() y _handleClientData():

Al aceptar un cliente, lo pasa rápidamente también a modo O_NONBLOCK usando fcntl y lo 
añade a _pollFds y a nuestra lista de instancias de Client.
Cuando un cliente habla (en _handleClientData()), intentaremos hacer un recv(). Si retorna 
0 o < 0, significa que el cliente se fue o falló, borramos el 
cliente y cerramos el socket. Si hay datos, los imprimimos por ahora.

Dudas técnicas:
🧠 1. ¿Qué es un socket y para qué sirve?
Un socket es como un "enchufe" o un puente de comunicación entre dos ordenadores a través de internet (o localmente).
En este proyecto, tu _serverSocket es el enchufe principal. Su única utilidad es hacer de recepcionista: se queda escuchando en el puerto 6667 y, cuando un cliente (Netcat o Irssi) "llama a la puerta", crea un NUEVO socket secundario (el clientFd en _acceptNewClient). Así, el socket principal vuelve a su trabajo de recepcionista, y el servidor usa el nuevo socket secundario de ese cliente para hablar y escuchar única y exclusivamente a esa persona.

🧠 2. ¿Qué hace fcntl?
fcntl significa File Control. En sistemas Unix, un socket es tratado como si fuera un archivo.
Por defecto, cuando intentas leer datos de un socket (recv) o aceptar un nuevo cliente (accept), si no hay nadie hablando o no hay nuevos clientes, tu programa se queda pillado/congelado (bloqueado) esperando hasta que llegue algo.
Como 42 nos prohíbe usar hilos o procesos paralelos (fork), si nos bloqueásemos esperando a un cliente, los demás clientes no podrían chatear.
Al usar fcntl(fd, F_SETFL, O_NONBLOCK);, le estamos diciendo al sistema operativo: "Pon este socket en modo NO BLOQUEANTE". Así, si intentamos leer y no hay nada, en lugar de congelarse, devuelve un error diciendo "no hay datos, sigue con tu vida".

🧠 3. ¿Por qué se declaran structs dentro de la función?
Vemos cosas como struct sockaddr_in o struct pollfd declaradas dentro de _initSocket() o _acceptNewClient(). Esto se hace así porque son estructuras de configuración temporal que nos exigen las funciones base del sistema operativo (C estándar).

sockaddr_in: La usamos sólo durante unas líneas para formatear la dirección IP y el puerto de la manera exacta y de bajo nivel que entiende la función bind() o accept(). Una vez hecho, ya no necesitamos recordar esa estructura.
Lo mismo para la variable temporal pollfd pfd;, sirve para empaquetar un FD y un "tipo de evento a vigilar" antes de guardarlo en nuestro vector de la clase.
🧠 4. La estructura del poll, su tamaño, y cómo sabe si es un cliente nuevo o datos
poll() bloquea el código hasta que al menos UN socket del chateo o del servidor hace algo.

El tamaño: El vector std::vector<struct pollfd> _pollFds; contiene una lista de todos los sockets que nos importan. El primer elemento de la lista (índice 0) SIEMPRE es el _serverSocket, y todos los que se van añadiendo después son clientes. Usamos _pollFds.size() para decirle a la función poll() exactamente a cuántos sockets tiene que vigilar.
Saber quién hace qué: Una vez poll() termina de esperar y nos devuelve control, recorremos con un for todo el vector. A cada entrada le preguntamos: "¿Ha ocurrido un evento en ti (revents & POLLIN)?".
Si ese FD que acaba de brillar coincide con nuestro _serverSocket, significa que el recepcionista tiene una llamada: tenemos un cliente nuevo (_acceptNewClient).
Si ese FD brillante NO es el del servidor, significa que algún cliente ya conectado ha pulsado Enter: Alguien mandó un mensaje (_handleClientData(fd)).

Client.cpp

Cada vez que recibimos algo de _handleClientData, el texto se almacena (appendBuffer()) en el interior de <Client>.
Como los clientes pueden mandar la mitad de la palabra y luego la otra (ej: com ... man... d\n) el servidor solo hará algo cuando ese cliente detecte un salto de línea \n en su buffer con nuestro nuevo método hasCompleteMessage().
Si detecta un salto de línea, extrae apropiadamente el texto completo (extractMessage()) borrando las basuritas finales como el \r propio del protocolo IRC y dejando solo el comando real, ej: "PASS mipassword123" o "JOIN #42".
Los imprime en un formato súper claro: Command from client 4: [USER Juan 0 * :Juan perez]. Y llama a la futura función _processMessage.
Esta extracción, al estar hecha con un modelo while (client->hasCompleteMessage()) protege de ataques en los que el cliente mande muchos comandos de golpe de una sola pulsación
